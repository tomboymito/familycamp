# FamilyCamp: скорректированный план доработки под ваш ТЗ

## 1) Что уже есть сейчас (чтобы не делать лишнего)

На текущем backend уже реализованы базовые API-эндпоинты:
- создание/получение бронирований;
- проверка доступности;
- изменение статуса брони;
- работа с типами размещения и гостями.

Также уже есть базовая модульная структура FastAPI, конфиг через env и CORS.

✅ Это значит: **фундамент есть**, не начинаем с нуля.

---

## 2) Что меняем в моем плане с учетом вашего ТЗ

Ваш принцип: **не ломать текущий сайт и дизайн**, делать только точечные доработки.

Поэтому обновленный подход:
1. Не трогаем визуальную часть страниц без необходимости.
2. Не делаем редизайн.
3. Frontend меняем только в зоне формы бронирования и пользовательских сообщений.
4. Основной объем работ переносим в backend + БД + CRM-интеграцию.

---

## 3) Целевая схема (как вы и предложили)

**Frontend сайта → Backend API → PostgreSQL → CRM**

Роли систем:
- Сайт: принимает заявку и показывает результат пользователю.
- Backend: проверка доступности, сохранение брони, синхронизация с CRM.
- CRM: дальнейшая работа менеджера с клиентом.

---

## 4) Сверка вашего плана: "есть / добавить"

### 4.1 Не ломать текущий сайт
- Статус: **берем как обязательное правило**.
- Действие: работаем точечно в текущем HTML/JS, без редизайна.

### 4.2 Форма бронирования
- Частично есть (модель брони и API уже есть), но нужно доработать поля под ваш список:
  - check-in/check-out,
  - тип места,
  - guests_count,
  - name/phone/email,
  - comment.
- Действие: привести frontend-форму и backend DTO к единому контракту.

### 4.3 Логика бронирования
- Частично есть: создание брони реализовано.
- Не хватает: корректной серверной проверки конфликтов дат и бизнес-правил для отказа/успеха.
- Действие: добавить строгую проверку пересечений и единые ответы API.

### 4.4 Проверка занятости
- Базовый endpoint есть.
- Не хватает: надежной защиты от двойного бронирования (в т.ч. при конкурентных запросах).
- Действие: транзакционная проверка + DB-ограничения/индексы + корректные статусы.

### 4.5 Статусы брони
- Сейчас есть похожие, но не полностью совпадают с вашими.
- Нужно внедрить набор:
  - `new`, `pending`, `confirmed`, `cancelled`, `crm_error`.
- Действие: миграция enum/справочника статусов и обновление API.

### 4.6 Backend API
- Уже есть часть методов.
- Нужно добавить/уточнить:
  - повторная отправка брони в CRM;
  - список броней для внутреннего использования с фильтрацией;
  - обновление статуса по новому lifecycle.

### 4.7 Защита формы
- Сейчас практически нет.
- Нужно добавить:
  - anti-spam (honeypot + rate limit),
  - серверную валидацию phone/email/required,
  - защиту от повторной отправки (idempotency key).

### 4.8 Сообщения пользователю
- Нужно стандартизировать 4 сообщения:
  - успешно отправлена;
  - даты недоступны;
  - ошибка отправки формы;
  - временная техническая ошибка.
- Действие: единый JSON-формат ответов backend + соответствующие тексты на frontend.

## 5) Обновленный план работ (по шагам)

## Этап A — Freeze UI и контракт формы (0.5 дня)
1. Зафиксировать: дизайн/верстка не меняются.
2. Зафиксировать JSON-контракт формы брони (поля из вашего ТЗ).
3. Согласовать названия полей между frontend и backend.

**Результат:** стабильный контракт без редизайна.

## Этап B — PostgreSQL-схема под бронирование + CRM (1–2 дня)
1. Перейти на PostgreSQL (managed).
2. Ввести таблицы (или эквивалент через существующие модели):
   - `spots`
   - `customers`
   - `bookings`
   - `crm_sync_log`
3. Добавить нужные индексы и ограничения.

**Результат:** БД полностью соответствует вашему процессу.

## Этап C — Бизнес-логика брони (1–2 дня)
1. Проверка пересечения дат по активным статусам.
2. Защита от двойного бронирования.
3. Новый lifecycle статусов: `new/pending/confirmed/cancelled/crm_error`.
4. Сохранение брони даже при ошибке CRM.

**Результат:** надежная логика заявки.

## Этап D — CRM интеграция (1–2 дня)
1. Отправка заявки в CRM после сохранения брони.
2. Логирование request/response/error в `crm_sync_log`.
3. API на повторную отправку в CRM.
4. Обновление `crm_sync_status` и статуса брони при ошибках.

**Результат:** прозрачная и управляемая интеграция.

## Этап E — API доводка под эксплуатацию (1 день)
1. Финализировать методы:
   - create booking,
   - check availability,
   - list bookings (internal),
   - resend to CRM,
   - update status.
2. Единые ответы и коды ошибок.

**Результат:** готовый внутренний API-контур.

## Этап F — Точечные frontend-изменения (0.5–1 день)
1. Подключить форму к API.
2. Показать 4 пользовательских статуса из ТЗ.
3. Добавить client-side базовую валидацию (но финальная проверка — серверная).

**Результат:** рабочий UX без редизайна.

## Этап G — Защита формы и устойчивость (0.5–1 день)
1. Anti-spam.
2. Idempotency от повторной отправки.
3. Rate limiting по IP/endpoint.

**Результат:** форма устойчива к спаму и дублям.

## Этап H — Deploy на VPS + Docker (1 день)
1. Backend контейнер.
2. Nginx для фронта/прокси.
3. Подключение к managed PostgreSQL.
4. Smoke-проверка end-to-end.

**Результат:** готовность к production-выгрузке.

---

## 6) Приоритет именно сейчас (следующий практический шаг)

Предлагаю начать с **Этапа B (БД и миграции)**, потому что:
- все остальные этапы завязаны на правильной структуре данных;
- статусы и CRM-лог лучше заложить сразу;
- потом API и frontend пойдут без переделок.
